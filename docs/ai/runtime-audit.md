# Runtime Audit

## Scope

This audit covers the current runtime structure visible in:

- `runtime/`
- `scripts/unified-generation-runtime.js`
- `scripts/runtime-generation-adapter.js`
- `scripts/runtime-quality-analyzer.js`
- runtime preview/report logic in `index.js`
- report-producing verification and simulation scripts under `scripts/`

This is an analysis-only document. It does not approve runtime behavior changes.

## Current runtime shape

The repository already has a recognizable runtime split:

- `runtime/execution/` handles local execution modes, sandbox execution, output sanitization, and response validation.
- `runtime/identity/` handles author identity scoring, persona memory, worldview, emotional signature, rhetorical patterns, continuity, and drift detection.
- `runtime/campaign-memory/` handles topic history, CTA history, narrative arcs, audience fatigue, campaign sequence, and campaign state persistence.
- `runtime/strategy/` handles strategic brain, trust building, authority pacing, emotional funnel, conversion pressure, audience state, positioning, narrative loop, and strategic memory.
- `runtime/editorial/` handles editorial direction, storytelling, format orchestration, audience temperature, attention loop, content balance, pacing, emotional arc, calendar slotting, and editorial memory.
- `runtime/stabilization/` handles prompt/output quality rules and shared scoring utilities.
- `scripts/runtime-generation-adapter.js` currently acts as the integration point between unified runtime state, retrieval/prompt assembly, stabilization, execution sandbox, identity, campaign memory, strategy, and editorial direction.
- `scripts/unified-generation-runtime.js` still acts as a broad orchestration prototype for cognition loading, local retrieval simulation, generation planning, content production, validation, and persistence.

## Boundary observations

### Runtime package boundaries are improving but not final

The `runtime/` folders are named by responsibility, which is good for future layered development. The main boundary weakness is that several important runtime orchestration responsibilities still live in `scripts/` instead of under a runtime-owned orchestration layer.

Examples:

- `scripts/unified-generation-runtime.js` owns runtime state creation, cognition loading, retrieval candidate loading, campaign planning, production pack construction, validation, quality scoring, and cognition persistence.
- `scripts/runtime-generation-adapter.js` owns prompt package assembly, local retrieval candidate creation, generation plan creation, runtime quality stabilization, execution sandbox wiring, identity runtime execution, campaign memory execution, strategic brain execution, editorial director execution, and final payload shaping.
- `scripts/runtime-quality-analyzer.js` imports `runtime/stabilization/` rules but remains in `scripts/`, making stabilization partly runtime-owned and partly script-owned.

Risk: future Codex sessions may treat `scripts/` as safe utility space and accidentally modify core runtime behavior.

### Retrieval and runtime are still coupled through adapter code

The runtime adapter imports retrieval and generation assembly helpers:

- `assembleContextPack`
- `createGenerationPlan`
- `rerankRetrievalItems`
- `assembleFinalPrompt`
- `createLocalRetrievalCandidates`

This makes the adapter responsible for both runtime orchestration and retrieval/prompt assembly. That is acceptable for the current admin-only preview pipeline, but it is a risky long-term boundary because retrieval behavior, prompt assembly, and runtime decisions can change together.

Risk: a change intended to improve runtime decisions could alter retrieval selection, prompt grounding, or output structure.

### Runtime memory systems are parallel but stateful

The following state stores are separate and mostly well-contained:

- `runtime/identity/persona-memory.js`
- `runtime/campaign-memory/campaign-state-store.js`
- `runtime/strategy/strategic-memory-store.js`
- `runtime/editorial/editorial-memory-store.js`
- cognition storage managed by `scripts/unified-generation-runtime.js`

The separation is good, but all memory systems write local JSON state under `storage/`. Parallel execution could create race conditions if multiple sessions run runtime previews for the same expert.

Risk: future parallel Codex runs can produce non-deterministic local state if they execute simulations that persist by default.

### Reports are generated by many owners

Reports are currently produced by simulation scripts, verification scripts, runtime preview code in `index.js`, and batch generation scripts. Report output locations include:

- `reports/runtime/`
- `reports/runtime-generation/`
- `reports/runtime-execution/`
- `reports/runtime-identity/`
- `reports/runtime-campaign-memory/`
- `reports/runtime-strategy/`
- `reports/runtime-stabilization/`
- `reports/runtime-editorial/`
- `reports/checks/`
- `reports/content-strategy/`
- `reports/content-production/`
- `reports/content-analytics/`
- `reports/cognitive-graph/`
- expert-local report folders under `expert_profiles/<expert_id>/reports/`
- `feedback_reports/`

Risk: report names and locations can drift faster than runtime architecture, which makes verification harder and creates merge conflicts in generated artifacts.

### Telegram runtime and admin runtime preview touch the same file

`index.js` remains the Telegram bot runtime file and also contains admin runtime preview/report functions. This creates a direct coupling between:

- production Telegram bot handlers
- admin-only runtime preview command
- runtime report serialization
- runtime adapter execution

Risk: even documentation-safe runtime work can become production-sensitive if it requires editing `index.js`.

## Duplicated responsibilities

### Runtime validation and quality scoring

Validation concepts appear in several places:

- `scripts/unified-generation-runtime.js` validates runtime state and calculates runtime quality.
- `scripts/runtime-quality-analyzer.js` evaluates prompt/runtime quality and stabilization metrics.
- `runtime/execution/runtime-response-validator.js` validates final execution output.
- `runtime/execution/runtime-sandbox.js` combines execution, sanitization, validation, and quality.
- verification scripts validate expected report files and safety constraints.

Recommendation: keep behavior unchanged now, but document a future distinction between:

- runtime decision validation
- prompt package quality analysis
- execution output validation
- verification/report validation

### Campaign, strategy, and editorial pacing

Pacing appears in multiple runtime layers:

- CTA and emotional pacing in `runtime/stabilization/`
- CTA history and audience fatigue in `runtime/campaign-memory/`
- authority, trust, conversion, and emotional funnel pacing in `runtime/strategy/`
- content freshness, audience temperature, attention decay, and saturation in `runtime/editorial/`
- trust and CTA pacing in `scripts/unified-generation-runtime.js`

This is not inherently wrong, but names must clarify whether a module observes memory, makes strategy decisions, or produces editorial recommendations.

### Report summaries

Runtime signal summaries are assembled in several places:

- adapter return payloads
- runtime preview JSON/Markdown in `index.js`
- verification reports in `scripts/verify-*.js`
- simulation reports in `scripts/simulate-*.js`

Recommendation: future standardization should define a single report schema vocabulary before moving code.

## Oversized or high-coupling files

These files deserve special caution:

- `index.js`: Telegram bot runtime plus admin runtime preview/report serialization.
- `scripts/unified-generation-runtime.js`: broad runtime prototype with state, retrieval simulation, planning, validation, and persistence.
- `scripts/runtime-generation-adapter.js`: main integration hub for runtime, retrieval, prompt assembly, stabilization, execution, identity, campaign memory, strategy, editorial direction, and final payloads.
- `runtime/editorial/editorial-director.js`: large but still layer-local; high fan-in from editorial sub-engines.
- `runtime/strategy/strategic-brain.js`: large but still layer-local; high fan-in from strategic sub-engines.
- `runtime/campaign-memory/campaign-memory-engine.js`: layer-local but stateful and persistence-sensitive.
- `runtime/identity/author-identity-engine.js`: layer-local but stateful and persistence-sensitive.

## Dangerous coupling

High-risk coupling points:

- `index.js` imports and executes `runRuntimeGenerationAdapter`.
- `scripts/runtime-generation-adapter.js` imports both runtime modules and retrieval/generation assembly helpers.
- `runtime/execution/` imports `scripts/runtime-quality-analyzer.js`, creating a runtime-to-scripts dependency.
- `scripts/unified-generation-runtime.js` imports content strategy, production, analytics, retrieval intelligence, generation orchestration, and author voice modules.
- runtime engines default to persistence unless callers opt out.
- report-generation locations are embedded directly in verification and simulation scripts.

## Architecture risks found

1. Runtime orchestration is split between `runtime/` and `scripts/`.
2. `scripts/runtime-generation-adapter.js` is the main integration bottleneck and future merge-risk file.
3. Report directories and naming are inconsistent enough to make automated verification brittle.
4. Runtime preview/report code in `index.js` keeps admin tooling close to production Telegram handlers.
5. Persistent local state can be mutated during simulations unless explicitly disabled.
6. Generated reports and state files are likely conflict points in future parallel sessions.

## Safe next actions

For future iterations, prefer documentation and interface mapping before code movement:

- Define runtime sublayer ownership.
- Define report categories and naming.
- Define which scripts are orchestration prototypes versus stable runtime interfaces.
- Mark persistence-sensitive simulation commands.
- Create a migration plan before moving any runtime adapter logic.

## Stop-condition notes

No production risk was introduced by this audit because no runtime code, retrieval behavior, Telegram UX, deployment config, secrets, or environment files were changed.
